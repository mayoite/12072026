# World-Standard Design Benchmark Protocol

## Mandate

The current planner UI is not the design baseline. The donor is a functional and behavioral reference only. The React planner requires a fresh design approach at the standard of leading current spatial-design, CAD, floor-planning, and professional creative tools.

Do not reproduce the donor's visual composition, panel arrangement, styling, or information hierarchy unless current evidence independently supports it.

## Mandatory benchmark agent

Before executing any phase or slice that changes UI, UX, drawing workflows, toolbars/panels, inventory presentation, visual design, onboarding, 2D/3D transitions, or export workflow:

1. Summon a dedicated benchmark/research agent.
2. Require current web research using direct product/help sources.
3. Compare at least five relevant leading products, including:
   - two floor-planning/interior-design products;
   - one CAD or spatial-authoring product;
   - one professional creative-tool interface;
   - one product strong in large-catalog or asset-library UX.
4. Review desktop and tablet behavior where public evidence exists.
5. Separate observed facts from recommendations.
6. Record dated citations and access date.
7. Save a new immutable advisory report:
   `REFERENCE-EXECUTION-BENCHMARK-<phase>-<YYYY-MM-DD>.md`.
8. The primary agent reviews the report, rejects imitation and weak suggestions, and incorporates accepted decisions into binding plans/design records before implementation.

Existing reference reports are background only and do not satisfy a future execution benchmark when materially stale.

## Required research dimensions

- first-run, empty state, templates, and progressive teaching;
- canvas-to-chrome ratio and visual hierarchy;
- drawing-tool discovery, feedback, snapping, measurements, and cancellation;
- selection, properties, contextual controls, command search, and shortcuts;
- constrained docking, movement, resizing, presets, reset, and responsive adaptation;
- inventory taxonomy, search ranking, filters, variants, previews, recents, favorites, replacement, and large-data behavior;
- 2D/3D continuity, loading, camera state, missing assets, and performance;
- visible save/offline/recovery/conflict truth;
- import/export lifecycle, preflight, progress, preview, history, and stale output;
- accessibility, keyboard operation, focus, contrast, and non-canvas project structure;
- visual language, density, typography, iconography, color, motion, and high-contrast behavior.

## Design output required before implementation

The primary agent must produce or update a binding phase design brief containing:

- target users and top tasks;
- information architecture;
- default desktop and tablet layouts;
- viewport capability statement;
- toolbar/panel grammar;
- command and drawing-tool hierarchy;
- inventory navigation model;
- key screen and state matrix;
- accessibility behavior;
- performance budgets;
- accepted inspirations with reasons;
- rejected inspirations with reasons;
- intentional differences from donor and competitors;
- testable acceptance criteria.

No UI implementation begins from screenshots alone or from an agent's unreviewed suggestions.

## Fresh-design acceptance

Acceptance requires:

- task completion and drawing correctness;
- clear, accessible, responsive structure;
- professional visual coherence;
- an excellent default layout before customization;
- inventory usable at scale;
- recoverable toolbar/panel customization;
- no donor visual debt carried forward without explicit justification;
- evidence that the result is internally coherent, not a collage of competitor patterns.

## Anti-copy rule

Borrow interaction principles and validated patterns, not proprietary colors, wording, icons, exact geometry, screen composition, assets, or trade dress.

## Stale-evidence policy

Immutable advisory reports (dated `REFERENCE-EXECUTION-BENCHMARK-*.md` files) must not be edited after creation. When source changes invalidate findings in an immutable report:

1. Do not edit the original file.
2. Add a dated correction note in the phase evidence file or HANDOVER.md referencing the original report and stating what changed.
3. If the stale findings affect acceptance decisions, create a new dated benchmark report under `DESIGN-BENCHMARK-PROTOCOL.md`.

This preserves the historical record while ensuring current decisions are based on current truth.
