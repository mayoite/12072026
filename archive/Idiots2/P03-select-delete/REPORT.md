### 3.2 Selection model (abstract)

| Behavior | Floorplanner pattern | O&O W3 adoption |
|----------|----------------------|-----------------|
| Click object | Select; sidebar becomes object properties | Hit-test furniture; selection drives props/delete affordance |

| Multi wall/room actions | remove, duplicate, rotate, mirror | Not W3 product bar |

### 3.3 Keyboard grammar (manual table — abstract keys)

Essential always-on keys from Floorplanner EN manual (as distilled in INSPIRATION):

| Function | Key(s) | Context | O&O W3 |
|----------|--------|---------|--------|
| Exit mode / deselect | **Esc** | Drawing or selection | **In W3 grammar** (clear selection) |
| Delete | **Del / Backspace** | Selection active | **Hard gate** |
| Undo / Redo | **Cmd/Ctrl+Z / Y** | Editor | **Undo hard**; redo exists but not W3 spine claim |
| Disable snap | Hold **S** | Draw/move | Not W3 |

| Shortcut help | **?** | Sidebar | P09 pattern |

**Pattern sentence:** Tool activation shortcuts + **always-on** Esc / Delete / Undo / Snap-off / Pan. Mode conflicts resolved by drawing mode vs selection mode.

## 13. Pure delete / single history / preventDefault / Esc

### 13.1 Why purity matters

If delete is N× `updateProject` in a loop:

- History gets N past frames for one user intent  
- One undo restores only one id  
- Multi-select (even accidental `ids: [a,b]`) becomes undo footgun  
- Expert false-reverse: “multi-history delete good enough” is a **buyer-facing lie**  

### 13.2 Pure API design (appendix)

```
applySelectionDelete(project, selection) → { project, selection }
  // or live variant: → project only, workspace clears selection
```

**Rules (binding):**

1. `type === "none"` or empty ids → **same project reference**  
2. Map types: wall→walls, door→doors, window→windows, furniture→furniture, room→rooms  
3. Remove **all** listed ids from active floor in **one** clone  
4. Skip locked; if nothing removed → same ref  
5. History identity: only push past when project reference (membership) actually changes  
6. Always clear selection after attempt path (workspace)  
7. No `any`  
8. Wall delete may cascade openings (shared path safety — not CP-03 product story)

### 13.3 Workspace wire

1. Read `workspaceCanvas.selection`  
2. Apply pure delete  
3. If project changed → **one** `updateProject`  
4. `setSelection({ type: "none", ids: [] })`  

Properties panel single-id `deleteEntityFromProject` OK for P03.

### 13.4 Keyboard cases (Task 04)

1. Delete → `deleteSelection` once  
2. Backspace → once  
3. Both **preventDefault** on cancelable events  
4. Ctrl+Backspace does **not** delete  
5. Editable `<input>` / contentEditable does not delete  
6. Omitted handler does not throw  

### 13.5 Esc (Task 06)

Cancel path must:

- Existing cancel (pending catalog, canvas cancel)  
- **Plus** `setSelection({ type: "none", ids: [] })`  

Editable fields still early-return on keyboard. Do not expand to full W8 chrome audit in P03.

### 13.6 Suggestions P0-4 — history identity

`applySelectionDelete` must return **identical** `project` reference when membership unchanged. A clone-no-op would pollute undo.

### 13.7 Suggestions P1-3 — single-history even for single-id

Today one id = one step is OK if single update. Two ids = two steps is the bug pure helper fixes. Multi-id pure tests are **mandatory** even if UI is single-select P0.
