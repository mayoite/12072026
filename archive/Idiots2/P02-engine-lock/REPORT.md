## 2. What “engine lock” actually means

### 2.1 Definition (operational)

**Engine lock** is a one-time freeze of:

1. **Which interactive 2D technology** owns furniture/structure interaction *destination* vs *interim*.
2. **Which 3D technology** owns the planner workspace (not admin single-SKU preview).
3. **What is forbidden** (hybrid multi-engine thrash, Unity, permanent dual-CAD).
4. **What is not an engine vote** (Block2D symbols, SVG publish pipeline, model-viewer admin, photoreal queue).
5. **Where flags live** and default production behavior.
6. **Fail-forward rule** if the destination engine is later proven unworkable (one full replacement, never hybrid).

It is **not**:

- Shipping Fabric full walls this week.
- Proving W3–W8.
- Upgrading packages.
- “Evaluating Konva just in case.”
- Claiming the product is shippable because packages are installed.

### 2.2 Why lock early (P02 before P03)

Historical thrash signals from research + product context:

| Thrash pattern | Cost |
|----------------|------|
| Canvas + Konva + Fabric simultaneous ownership | Pointer wars, dual hit-test, months of half-engines |
| “Fabric is only insurance” → delete slice | Violates Plan A destination; reopens years of object-CAD debt |
| “Switch to Babylon/PlayCanvas for better mesh” | Mesh quality is **content + LOD**, not engine brand |
| “Disable orbit to stabilize tests” | Product without navigate camera fails every 3D winner bar |
| Re-debate engines mid W-gate | Agents burn context; no buyer proof |

P02 exists so later phases **only ship product gates** on a fixed path.

### 2.3 CP-02 vs product green

| State | Meaning |
|-------|---------|
| **CP-02 green** | Written lock + flags + entrypoints + package pins + anti-thrash + owner sign-off (or explicit deferral) under `results/.../01-engine-lock/` |

## 6. O&O live engine truth (architecture ideas)

### 6.1 Document-first architecture (the real “engine”)

Every winner and every good open planner converges here:

```
Document model
  walls, openings, furniture
  UUIDs, mm, poses, catalog SKU refs
        │
        ├── 2D view(s): paint + hit-test + tools
        ├── 3D view(s): extrude + place meshes + camera
        └── Persist: IDB → later cloud
```

**If 2D and 3D disagree on IDs, you do not have continuity — you have two demos.**

Lock engines **around** the document; never couple storage shape to a specific renderer’s JSON (e.g. do not make Fabric serialized scene the source of truth).

### 6.2 Locked architecture (Approach A)

```
Document (UUID entities, mm)
  ├── 2D destination: Fabric.js v7 full stage
  │     flag path: canvas-fabric-stage/  (default OFF)
  ├── 2D interim (live): FeasibilityCanvas (Canvas 2D API)
  ├── 3D: Three.js + R3F path + OrbitControls ON by default
  ├── Catalog: O&O SKUs + Block2D + modular mesh
  └── Persist: IDB first; cloud honesty later
```
