## 1. What global-standard manufacturer planners expect at “foundation / unlock”

**Product bar — not process ceremony.** At foundation (before “journey green” claims), a serious manufacturer furniture planner program locks **what the product is** and **what evidence will mean**, so later PASSes cannot drift into paperwork theater.

### 1.1 Buyer-facing product contract (must be named, not assumed)

A facilities / workplace buyer journey is the unit of foundation, not “agents may commit.”

| Expectation | Manufacturer-planner meaning | O&O north-star mapping |
|-------------|------------------------------|------------------------|
| **Who** | Non-developer buyer / facilities / dealer | Facilities buyer, no developer |
| **What they do first session** | Open plan tool → structure → place **our** SKUs → read plan | Walls/door + O&O-scale furniture on `/planner/open3d` or guest |
| **2D plan language** | Readable symbols at print-ish zoom; system families, not random free models | Block2D **and/or** published SVG authority must be **decided and named** |
| **3D language** | Parts readable enough to trust configuration (toe/carcass/door class), not photoreal | Modular mesh bar, not apology boxes |
| **Edit loop** | Select · move · delete · undo without leaving the canvas | W3 product bar |
| **Return tomorrow** | Project survives reload; status does not lie about cloud | W5 + W6 honesty |
| **Money path** | Dimensions + configuration feed quote/BOQ later | BOQ/quote **path** > photoreal (already in engine lock text; not product-done) |

Foundation does **not** require full multi-client factory, priced BOQ product, Fabric cutover, or cloud multi-tenant. It **does** require the above contract written so agents cannot redefine “done” as unit-green alone.

### 1.2 Asset & catalog authority (hard product foundation)

Manufacturer planners treat **catalog truth** as infrastructure, not a later nice-to-have:

1. **SKU / system model** — family → shape → size grid → modules (already in product context; must be foundation vocabulary).  
2. **2D authority named** — which path draws the plan mark:
   - procedural Block2D on FeasibilityCanvas, **and/or**
   - published SVG files under a public catalog path, **and/or**
   - Fabric object later.  
3. **3D authority named** — modular generator vs GLB stamp; what “good enough” means (readable parts, not Homestyler).  
4. **Honesty rule** — never claim portal/admin SVG publish proves canvas symbols unless wired end-to-end.  
5. **Scale of assets** — foundation admits **inventory debt**: how many real symbols/files exist vs demo placeholders.

**Repo truth today:** `site/public/svg-catalog/` has **4** SVGs; open3d plan canvas draws **Block2D**, not those files (`05-symbols-svg/04-svg-honesty/NOTES.md`). Foundation that ignores this is theater.

### 1.3 Engine + metric freeze (product, not checkbox cosplay)

Already partly present in Approach A / CP-02 language — foundation expects:

| Lock | Global-standard reason |
|------|------------------------|
| One **interim** 2D + one **destination** 2D | Stop thrash mid-buyer journey |
| 3D = orbit-on viewer, not a second plan editor | Buyer inspects configuration |
| Catalog = manufacturer / O&O systems | Not free GLB dump |
| Success = plan accuracy + place + save + **quote path** | Not photoreal race |

### 1.4 Language discipline (foundation artifact)

| Word | Must mean |
|------|-----------|
| **UNLOCK / W0 GATE PASS** | Permission + approach + ethics path + no worktree + evidence note |
| **FOUNDATION COMPLETE (product)** | Buyer contract + authority map + residual inventory + dual-language scoreboard frozen |
| **W-gate PASS** | That gate’s criteria met with **fresh-enough** path evidence for claim scope |
| **PRODUCT FINISHED / SHIP** | North star for buyer end-to-end — **never** inferred from CP-00…10 pack alone |

Global standard: **two scoreboards**, always. One process/gate, one product residual.

### 1.5 Explicit non-goals of foundation (so scope stays honest)

- Photoreal / Homestyler arms race  
- Full Fabric walls cutover before journey green (Approach A)  
- Multiplayer / CRDT / AR  
- CRM/SSR expansion as substitute for canvas usability  
- Competitor assets or brand paste  
- Claiming “complete” because CHECKPOINTS.md says PASS

## 6. Approach options for raising the foundation bar

### Option 1 — **Relabel only** (minimal)

**What:** Keep CP-00 criteria as process unlock. Globally rename language: `PASS (unlock)` vs product residuals. Rely on existing `HONEST-STATUS.md` / `TRUTH-LOCK.md`.

| Pros | Cons |
|------|------|
| Fast; no plan rewrite depth | Does not force authority map / asset census |
| Matches what unlock already is | Owner still sees thin foundation; easy to regress to bare PASS |
| Low agent thrash | Does not bind CP-05 SVG expectation |

**When OK:** If only confusion was word choice and owner accepts process-thin W0 forever.

### Option 2 — **Foundation addendum on CP-00** (recommended)

**What:** Keep process criteria (P1–P5). Add product foundation criteria (F1–F7) via addendum to `00-START.md` / CHECKPOINTS + evidence under `results/planner/global-standard-revision/CP-00/`. Historical unlock stays `PASS (unlock process)`. New status `PASS (foundation product)` only when F-criteria path-proven. Later phases inherit dual language + authority map.

| Pros | Cons |
|------|------|
| Directly fixes “PASS sold as complete” at the root | Requires head SYNTHESIS + plan edit (not this seat) |
| Path-provable; no fake product features in W0 | Slightly more ceremony — **but** ceremony that buys honesty |
| Forces SVG/Block2D/save/mesh vocabulary early | Agents must update evidence when census changes |
| Aligns with GLOBAL-STANDARD-REVISION program | — |

**When OK:** Default for this owner ask (bar too low; SVG debt; half-open product).

### Option 3 — **Merge W0 with CP-01 product truth** (heavy)

**What:** Collapse foundation into an expanded CP-01: deep live inventory + authority + residual becomes the real unlock for product claims. CP-00 remains pure permission only.

| Pros | Cons |
|------|------|
| Product truth is where inventory already lives | CP-00 stays thin; owner “foundation” label still weak |
| Avoids duplicating P01 inventory | Risk of “unlock done, truth later” gap repeating |
| Good if P01 is always re-run on tip | Does not alone fix GATE language across CP-02…10 |

**When OK:** If owner refuses any CP-00 product criteria and wants all substance in P01.

### Recommendation

**Option 2 — Foundation addendum on CP-00.**

Reasons:

1. Owner failure mode is **language + bar**, not missing permission.  
2. Asset/authority confusion (SVG vs Block2D) must be frozen **before** CP-05 is re-raised, not discovered mid-PASS.  
3. Option 1 is necessary but not sufficient.  
4. Option 3 dumps the hard work into CP-01 and leaves W0 as the same oversell vector.  
5. Option 2 matches the revision program’s dual-expert → synthesis → path-provable raised PASS pattern without inventing a second kill order.

**Head next (out of this seat):** UI-EXPERT → SYNTHESIS → plan addendum; do not implement product features under W0.
